## 1. Project description (elevator)

**Short (1–2 sentences):**
> SNAP Notice Navigator turns a confusing SNAP letter into plain language and a safe, source-backed checklist of
> next steps. It's not a chatbot and not a resource directory — it's a focused notice interpreter that reads a
> specific letter, explains it, and helps the user take the next step with confidence, while making clear that
> their state agency makes the final decision.

**Longer (paragraph):**
> Tens of millions of people rely on SNAP, but the system communicates through dense, legalistic notices —
> denials, document requests, recertification reminders — that are easy to misread. Most benefits aren't lost
> because people don't qualify; they're lost because a notice wasn't understood or a deadline was missed. SNAP
> Notice Navigator attacks that exact failure point. A user uploads or pastes their notice, reviews and corrects
> the text the app read, answers a few calm questions, and receives a plain-language explanation, the likely
> issue, a prioritized next-steps checklist, the documents to gather, and links to the official USDA and state
> pages each suggestion is based on — plus how to reach a real person. The AI interprets and suggests; the user
> decides; the agency makes the official call.

## 2. AI architecture explanation

> The app runs a modular A→G pipeline: ingestion → extraction (PDF text via pdf-parse, OCR via tesseract.js) →
> a human-in-the-loop review gate → classification + field extraction → guided intake → retrieval over a curated
> knowledge base → source-grounded recommendation → structured results.
>
> Two LLM steps use Anthropic's Claude (claude-sonnet-4-6) with **structured outputs** (Zod schemas +
> messages.parse), so every model response is schema-validated and deterministic in shape. Step one classifies
> the notice into one of six types and extracts fields (agency, dates, deadline, reason, requested documents,
> contact) with a confidence score. Step two generates the explanation and prioritized steps — but only from
> notice text plus snippets retrieved from our curated knowledge base.
>
> Crucially, it's a **hybrid**: a deterministic rules + regex layer always runs, providing an offline-safe
> fallback and backfilling any field the model leaves blank. **Source citations are attached from the knowledge
> base, never generated by the model**, so links are always real. If the API key is absent or a call fails, the
> app degrades to rules + templates and still produces a complete, source-backed result.

---

## 3. Human-in-the-loop statement

> The interface makes three roles explicit on both the landing and results pages: **the tool explains and
> suggests, the user decides, and the state SNAP agency makes the official decision.** Before any analysis runs,
> the user must review the exact text the app extracted and actively confirm "I've read this and it matches my
> notice" — separating document extraction from interpretation with a deliberate checkpoint. Every recommended
> action is framed as an option, not a command, and the user is always pointed to a real person (211, legal aid,
> their local office) for anything uncertain or time-sensitive.

---

## 4. Responsible AI guardrail

> The primary risk is incorrect interpretation or user over-reliance, and our mitigations are visible in both
> the UX and the code. We use hedged language only — "may", "likely", "possible" — and never claim guaranteed
> eligibility, that a decision is wrong, or that benefits are coming; this is enforced in the model's system
> prompt **and** in the deterministic templates. Every result carries a confidence band, with the model's
> reasoning available in a transparency panel; when confidence is low, the app says so plainly and routes the
> user to their local office or a benefits advocate. Each recommendation shows the official source it's based on,
> a persistent disclaimer states the agency is the final decision-maker, and we keep scope honest — deep support
> for three states, clearly-labeled federal-baseline guidance everywhere else. No accounts and no database:
> notice text stays in the browser session and uploaded files are processed in memory and discarded.

---

## 5. Data / tools disclosure (template)

> **AI model:** Anthropic Claude `claude-sonnet-4-6`, called server-side via the official Anthropic TypeScript
> SDK using structured outputs. Used for (1) notice classification + field extraction and (2) source-grounded
> recommendation generation.
> **OCR:** tesseract.js (on-device, open source). **PDF text extraction:** pdf-parse.
> **Knowledge base:** hand-curated snippets and links to official USDA/FNS and state SNAP pages (California,
> Texas, Missouri); source links come from this curated set, not from the model.
> **Frameworks:** Next.js 16, React 19, TypeScript, Tailwind CSS v4; schema validation via Zod.
> **Data handling:** no user accounts, no database, no analytics. Notice text is stored only in the browser
> session (sessionStorage); uploaded files are processed in memory and never persisted.
> **Sample data:** the three demo notices are synthetic and contain no real personal information.
